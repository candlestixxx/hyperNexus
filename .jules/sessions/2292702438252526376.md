# Project Architecture & Patterns Summary: "Super AI Plugin" (HyperNexus)

## 🏗️ Core Architecture
The project is a **local-first, TypeScript-based monorepo** managed with `pnpm` workspaces. It aims to be a comprehensive AI orchestration hub, providing CLI, TUI, and WebUI interfaces for managing AI models, tools, and workflows.

### Monorepo Structure (`packages/`)
- **`core/`**: The backbone service. Handles database persistence, MCP client/server logic, agent execution, and session management. Runs on Node.js using Fastify/Hono and Socket.io.
- **`ui/`**: A React/Vite-based dashboard for visualizing system status, managing MCP servers, inspecting tool calls, and configuring agents.
- **`cli/`**: A Commander-based CLI (`super-ai`) for interacting with the Hub.
- **`adapters/`**: Integration layers for various LLM providers (Anthropic, Gemini, OpenAI via Ollama).
- **`vscode/`**: A VSCode extension client that connects to the Hub.
- **`browser/`**: A Chrome extension for capturing web context and recording memories.

## 💾 Data & Persistence
- **Database**: Uses **Drizzle ORM** with **SQLite** (`better-sqlite3`) as the primary local storage.
- **Schema**: Defined in `packages/core/src/db/mcp-admin-schema.ts` (aggregating `hypernexus`, `council`, `mcp-browser`, and `mcp-tool-chain` schemas).
- **Repositories**: Follows a repository pattern (e.g., `ToolsRepository`, `ToolSetsRepository`) located in `packages/core/src/db/repositories/`.
- **Vector Store**: Uses **LanceDB** for semantic search and RAG capabilities, abstracted via `VectorStore`.

## 🤖 AI & Agent Orchestration
- **AgentExecutor**: The primary engine for running agents. It utilizes `ModelGateway` to abstract provider differences and supports session persistence.
- **MCP (Model Context Protocol)**: Central to the project. It acts as both an MCP client (connecting to external servers) and potentially an MCP server. It handles dynamic tool discovery, grouping, and reranking.
- **Memory Systems**:
    - **Native Memory**: Persistent JSON-based fuzzy search.
    - **Passive Memory**: `TrafficObserver` extracts facts from Hub interactions using LLMs.
    - **Document RAG**: PDF/Text ingestion via `DocumentManager`.

## 🛠️ Key Patterns & Decisions
- **Lazy Loading & Hooks**: MCP servers are initialized lazily. The system emits `pre_tool_call` and `post_tool_call` hooks for internal extensions.
- **Hydration**: Repositories often use a `hydrate` method to populate complex objects from flat DB rows (subject to optimization, as seen in the N+1 query task).
- **Session Handoffs**: Persistent session state is saved in `handoffs/` to allow resuming work across different interfaces or restarts.
- **System Doctor**: A dedicated service (`SystemDoctor`) ensures external dependencies like Node, Git, and Docker are present.
- **Environment Management**: Configuration is managed via `.env` files and `Profiles`, allowing easy switching between different setups.

## 🚀 Vision & Roadmap
