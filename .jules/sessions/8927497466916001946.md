### Project Architecture: Super AI Plugin (Hypercode) Monorepo

The **Super AI Plugin** is a comprehensive AI ecosystem organized as a TypeScript monorepo using **pnpm workspaces** and **Turbo**. It serves as an ultimate dashboard and coding harness for AI agents, MCP (Model Context Protocol) servers, and multi-model orchestration.

#### Core Monorepo Components
- **`packages/core`**: The backend service layer (Node.js/Fastify) managing the agentic loop, skills, memory (VectorStore/Fuse.js), and MCP proxies. It uses Drizzle ORM with SQLite for local-first persistence.
- **`packages/ui`**: A React/Vite dashboard providing a centralized interface for tools, billing, settings, and agent monitoring.
- **`apps/hypercode-extension`**: A sophisticated Chrome extension that bridges web-based AI interfaces (ChatGPT, Gemini, etc.) with the local core services.
- **`packages/cli`**: The `super-ai` command-line utility for interacting with the Hub.
- **`packages/adapters`**: Standardized integration layers for external LLM providers.

### Hypercode Extension: Content Script Architecture

The content script in `apps/hypercode-extension/pages/content` is built for high resilience and modularity:

#### 1. Unified State Management (Zustand Root Store)
- The extension has transitioned from fragmented stores to a **Unified Root Store** (`pages/content/src/stores/root.store.ts`).
- It uses **State Slices** (`UISlice`, `ConfigSlice`, `ToolSlice`, etc.) to isolate domain logic while maintaining a single source of truth.
- A **Backward Compatibility Wrapper** (`createSliceWrapper`) allows legacy code to use hooks like `useAppStore` while actually accessing the unified slices.

#### 2. Resilience and Orchestration
- **`MainInitializer`**: A central orchestrator that manages a strict initialization sequence (Environment -> Event Bus -> Core Services -> Stores -> Plugins -> Analytics).
- **Circuit Breaker Pattern**: Critical services and external integrations are wrapped in circuit breakers to prevent one failure from crashing the extension.
- **Performance Monitoring**: Granular timing of the initialization sequence is reported via the `eventBus`.

#### 3. Versioning and Environment Patterns
- **Dynamic Versioning**: The project standardizes on retrieving the extension version via `chrome.runtime.getManifest().version` with a `'0.0.0'` fallback for non-extension environments (e.g., tests).
- **Environment Management**: Distinguishes between development and production modes using a dedicated `env` package and `@dotenvx/dotenvx`.

#### 4. UI Isolation (Shadow DOM)
- Content script UI components are injected into host pages using **Shadow DOM**. This ensures CSS isolation, preventing the extension's styles from leaking into the host site and vice-versa.
- Uses a shared Tailwind configuration across the monorepo for consistent styling.

#### 5. Build and Tooling
- **Vite with `withPageConfig`**: Build settings for various extension pages are standardized through a shared configuration utility in `packages/vite-config`.
- **Turbo Orchestration**: The monorepo uses Turbo to cache and parallelize builds across the different packages.

### Design Decisions
- **Local-First with Remote Capability**: While the core runs locally, the architecture is designed to manage both local and remote (cloud) development sessions seamlessly.
- **Feature Parity Goal**: The project explicitly aims for feature parity with major AI coding harnesses (Claude Code, OpenCode, Codebuff) while offering a more modular, dashboard-centric experience.
- **Passive Memory and Traffic Observation**: Implements a `TrafficObserver` that extracts facts from LLM interactions to populate a passive memory system.
- **Redundancy and Fallbacks**: Focuses on automatic model switching and quota management to ensure continuous operation when provider limits are reached.
